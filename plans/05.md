## Plan: Phase 5 Coach Experience

Implement Phase 5 as a frontend-led coach dashboard on top of the already-complete Phase 03 coach APIs, with one concrete backend contract alignment and a spec refresh so the phase matches the shipped code and later-phase requirements. The implemented approach keeps the coach view read-only, reuses the member-page and API-client patterns, surfaces decision reasoning and escalation source clearly, exposes phrasing provenance to coaches, and adds an explicit in-app navigation path between member and coach views.

**Steps**

1. Phase A — Re-baseline the Phase 5 spec in `docs/phase-05-coach-experience.md` so it matches the implemented backend and adjacent phases. Phase 5 is a read-only coach UI over `GET /api/coach/nudges` and `GET /api/coach/escalations`; escalations remain `open` only, both feeds are ordered by `created_at DESC`, and recent nudges surface `matched_reason`, raw confidence, latest member response, escalation recommendation, and `phrasing_source`.
2. Phase B — Apply the backend alignment for coach visibility. Extend the coach nudge response to include `phrasing_source` from the `nudges` table while keeping ordering, default limits, and read-only behavior unchanged. This was the only backend contract change made for Phase 5.
3. Phase C — Extend the client contract layer to match the backend. Add typed coach response models and fetch helpers in `client/src/types/member.ts` and `client/src/api/client.ts`, following the existing `request()` pattern and keeping UI-only derivation limited to presentation helpers such as confidence bands and source labels.
4. Phase D — Add explicit frontend artifacts for the coach flow. Implement `client/src/pages/CoachPage.tsx` as the owner of coach-page fetching, loading, error, and empty-state logic, and split list rendering into read-only components `client/src/components/CoachNudgesList.tsx` and `client/src/components/CoachEscalationsList.tsx`.
5. Phase E — Add routing and in-app navigation for the review flow. Update `client/src/App.tsx` to expose `/coach`, preserve the current `/member` redirect behavior, add a `Coach dashboard` link from `MemberPage.tsx`, and add a `Member view` link back from `CoachPage.tsx` so reviewers can move between surfaces without editing the URL.
6. Phase F — Build the coach dashboard as two adjacent read-only views. Render open escalations in the left column and recent nudges in the right column on laptop widths, falling back to a stacked layout on smaller screens. The escalations queue emphasizes `source` so `member_action` cases stand out from rule-engine or low-confidence escalations. The recent nudges feed displays member name, nudge type, content, explanation, `matched_reason`, raw confidence plus a High/Medium/Low band, status, latest action, escalation recommendation, timestamp, and `phrasing_source`.
7. Phase G — Keep the UI intentionally lightweight and section-oriented. Each list owns its own loading, retryable error, and empty states so one failed request does not blank the whole coach page. The implementation stays read-only and does not add assignment workflows, resolution workflows, action-history drawers, filters, or audit dashboards.
8. Phase H — Verify the phase end to end. Re-run the backend API test script with the updated coach assertions, run frontend lint and build checks, and manually confirm seeded history, `ask_for_help` visibility, support-risk readability, explicit empty states, and the in-app route path between member and coach views.

**Decisions**

- Included scope: frontend coach dashboard, Phase 5 spec update, coach-side phrasing-source visibility, and explicit in-app navigation between member and coach views.
- Excluded scope: coach assignment/resolution workflows, analytics, audit dashboards, rich filtering, and full action history exploration.
- Preferred implementation posture: keep Phase 5 read-only and rely on existing Phase 03 APIs unless there is a narrow, justified contract gap.
- Implemented alignment change: expose `phrasing_source` in coach nudges so Phase 05 stays coherent with Phase 06 and Phase 08 verification.
