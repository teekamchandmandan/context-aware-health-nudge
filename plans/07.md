## Plan: Phase 7 Observability and Audit

Wire the existing audit/logging foundation into the remaining nudge lifecycle paths so the backend can explain what happened, why it happened, and whether phrasing came from the LLM or templates. The recommended approach is to keep this phase narrowly backend-focused: reuse the existing `audit_events` table and `app.audit` helpers, add structured JSON logging at decision/action boundaries, and prove the behavior with focused engine/API tests rather than adding any dashboard or analytics helper.

**Steps**
1. Phase A: lock the instrumentation contract and keep scope tight. Reuse the five stable event names from the Phase 7 spec (`nudge_generated`, `user_action`, `escalation_created`, `llm_call`, `llm_fallback`), confirm that raw signal intake is excluded from durable audit scope, and keep coach/member APIs unchanged.
2. Phase B: instrument durable audit writes in the decisioning path. In `server/app/engine.py`, add `nudge_generated` immediately after a nudge row is created with payload fields from `NudgeCandidate` plus the persisted `nudge_id`; add `escalation_created` when low-confidence or escalation-recommended candidates create an escalation; ensure these writes happen before the same transaction commit that persists the nudge/escalation rows. This step blocks Step 4 test coverage for generation flows.
3. Phase C: instrument durable audit writes in the member action path. In `server/app/main.py`, add `user_action` after the nudge status transition and action row insert, including `previous_status`, `new_status`, and `action_type`; for `ask_for_help`, also add `escalation_created` for the member-action escalation source. This can proceed in parallel with Step 2 once the payload contract is fixed, but both must finish before full API verification.
4. Phase D: finish structured runtime logging. Reuse `log_structured_event()` from `server/app/audit.py` for the same lifecycle boundaries: nudge generation, escalations, user actions, and existing LLM events. Configure JSON log emission in `server/app/main.py` `lifespan()` with stdout-friendly `logging.basicConfig(...)`, use `INFO` for normal workflow transitions, and keep `WARNING` for fallback/error-like cases already present in `server/app/phrasing.py`. This depends on Steps 2 and 3 so the logged payloads match the durable audit payloads.
5. Phase E: reconcile LLM observability rather than expanding it. Review `server/app/phrasing.py` to confirm the existing `llm_call` and `llm_fallback` payloads match the Phase 7 minimum fields, keep support-risk escalations deterministic, and avoid introducing any new audit event types or metrics endpoints. This is parallel with Step 4 but should finish before test expectations are finalized.
6. Phase F: add focused regression coverage. Extend `server/test_engine.py` to verify `nudge_generated` and rule-engine `escalation_created` rows are written with compact payloads. Extend `server/test_api.py` to verify `user_action` rows for `act_now`, `dismiss`, and `ask_for_help`, plus the member-action `escalation_created` row for `ask_for_help`. Include at least one LLM fallback/success assertion only if current tests do not already prove the existing phrasing instrumentation. This depends on Steps 2, 3, and 5.
7. Phase G: run manual verification against the prototype flows. Generate a nudge, perform each action type, trigger a low-confidence escalation path, trigger an LLM fallback path, and inspect SQLite `audit_events` plus stdout logs to confirm the sequence is reconstructable without noisy or redundant payloads. This depends on all prior steps.

**Decisions**
- Included: durable audit events for the five Phase 7 event types, stdout structured logging, and regression coverage for rule-engine and member-action paths.
- Excluded: signal-ingestion audit events, review-metrics helpers/endpoints, dashboards, external observability sinks, retention management, and alerting.
- Preserve current product contracts: coach pages continue reading `nudges` and `escalations`, while `audit_events` remains a backend review artifact.
- Keep payloads compact and stable: summary fields only, no large request/response bodies, and no expansion into analytics-style dimensions.

**Further Considerations**
1. If implementation shows repeated audit/log payload assembly in multiple places, add one small shared helper in `server/app/audit.py`; otherwise keep the wiring inline to avoid unnecessary abstraction.
2. If existing phrasing tests already cover `llm_call` and `llm_fallback` sufficiently, do not duplicate them in both test files; add only the missing assertions needed to make Phase 7 credible.
3. If transaction ordering becomes ambiguous during implementation, prefer writing audit rows in the same transaction immediately before `conn.commit()` so product state and audit history stay consistent.
