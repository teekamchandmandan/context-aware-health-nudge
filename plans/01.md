# Plan: Phase 01 â€” Foundation and Data

Set up the repo structure, FastAPI backend shell, SQLite schema for 6 tables, and deterministic seed data covering the baseline context for all 3 assignment scenarios. No decision logic, no UI, no LLM work.

## Decisions

- **Client scaffold**: Scaffold Vite+React+Tailwind in `client/` during Phase 1 via `npm create vite@latest`. Add Tailwind v4 config. No UI components â€” just the working shell so Phase 4 jumps straight to building the member page.
- **Dev-only reset endpoint**: Expose `POST /debug/reset-seed` guarded by `DEBUG=true` env var. Route is not mounted when `DEBUG` is unset/false.
- **Relative timestamps** in seed data (e.g., `utcnow() - timedelta(hours=6)`) so scenarios always satisfy trigger windows regardless of when seed runs.
- **`generated_by`** = `rule_engine` and **`phrasing_source`** = `template` on all seeded nudges.
- **`uuid4().hex`** for auto-generated IDs; member IDs are the stable text constants from the spec.
- No migration tooling â€” idempotent `CREATE TABLE IF NOT EXISTS` DDL is sufficient per spec.

   - `name` TEXT NOT NULL
   - `goal_type` TEXT NOT NULL

   **signals**
   - `id` TEXT PRIMARY KEY
   - `member_id` TEXT NOT NULL â†’ FK members
   - `signal_type` TEXT NOT NULL (`meal_logged`, `weight_logged`, `mood_logged`)
   - `payload_json` TEXT
   - `created_at` TEXT NOT NULL

   **nudges**
   - `id` TEXT PRIMARY KEY
   - `member_id` TEXT NOT NULL â†’ FK members

   - `escalation_recommended` INTEGER (bool)

   - `phrasing_source` TEXT NOT NULL DEFAULT 'template'
   - `created_at` TEXT NOT NULL

   **nudge_actions**
   - `id` TEXT PRIMARY KEY
   - `nudge_id` TEXT NOT NULL â†’ FK nudges
