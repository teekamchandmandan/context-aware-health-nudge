## Plan: Phase 6 LLM Safety and Phrasing

Implement Phase 6 as a tightly bounded backend phrasing layer that sits after deterministic nudge selection and before the final active-nudge response is returned. The improved plan keeps all decision ownership inside the existing rule engine, preserves current member and coach flows, uses deterministic template copy as the default path, and treats LLM phrasing as an optional upgrade that must fail closed. The implementation should also close a few important gaps the original draft left open: when phrasing should not run at all, how repeated reads avoid rephrasing churn, how fallback reasons stay queryable for later audit work, and how the phase stays testable without requiring a live provider.

**Gaps Closed In This Revision**

1. Clarifies that LLM phrasing runs only for newly created member-visible active nudges and does not run for existing active nudges on repeated reads, escalated outcomes, or `no_nudge` states.
2. Makes idempotency explicit: `GET /api/members/{member_id}/nudge` should not trigger a fresh provider call once an active nudge already exists.
3. Aligns fallback reasons with later audit expectations by standardizing `missing_key`, `timeout`, `provider_error`, `invalid_json`, and `validation_failure`.
4. Makes the prompt boundary stricter by excluding raw notes, image payloads, and full signal history unless reduced to a short structured summary first.
5. Adds a concrete testing posture that relies on a fake provider seam or mocked HTTP client rather than real OpenAI calls in automated tests.
6. Resolves the support-risk ambiguity: escalated outcomes remain deterministic and do not add member-visible LLM phrasing.
7. Calls out the need to align backend fallback templates with the Phase 6 spec so the plan does not silently preserve older template copy that conflicts with the docs.

**Steps**

1. Phase A — Re-baseline the Phase 6 implementation boundary in `docs/phase-06-llm-safety-and-phrasing.md`, `docs/plan.md`, and the code comments around the engine hook so all three say the same thing: the LLM can rewrite phrasing only after a nudge has already been selected, and it must never change `nudge_type`, `confidence`, `matched_reason`, `escalation_recommended`, `generated_by`, or the `active` vs `escalated` decision.
2. Phase B — Normalize deterministic fallback copy and policy constants. Extract the default templates, blocked terms, max-length limits, and 3-second timeout into an explicit backend policy module or phrasing module so the fallback path is centralized, easy to review, and aligned with the Phase 6 doc examples rather than the current older engine strings.
3. Phase C — Add the minimum backend configuration surface in `server/app/config.py`: `OPENAI_API_KEY`, an optional model name, and a phrasing timeout default. Keep configuration intentionally small and avoid introducing multi-provider or environment-specific abstraction for this prototype.
4. Phase D — Create a focused phrasing service in `server/app/phrasing.py`. The service should accept only the minimal structured context required by the spec: `nudge_type`, member goal, concise matched reason, concise explanation basis, and desired tone. It should call OpenAI through the existing `httpx` dependency, request JSON-only output, enforce the 3-second timeout, and return either a validated `{content, explanation}` payload or a typed fallback reason. This step depends on steps 1-3.
5. Phase E — Add hard validation before any LLM output can be persisted. Reject responses when JSON parsing fails, `content` exceeds 160 characters, `explanation` exceeds 160 characters, blocked terms appear, required fields are missing, or values are empty after trimming. On every rejection path, preserve deterministic templates and record the fallback reason. This step depends on step 4.
6. Phase F — Hook phrasing into the current engine flow only for new active nudges. The engine should first persist the deterministic active nudge with `phrasing_source = "template"`, then optionally upgrade it in place to `phrasing_source = "llm"` if and only if the provider succeeds and validation passes. Repeated reads of an existing active nudge should return the stored row without another LLM call. Escalated outcomes should stay on the current deterministic path and should not create member-visible phrasing. This step depends on steps 2-5.
7. Phase G — Add a small audit/logging helper rather than scattering one-off SQL writes. Use the existing `audit_events` table to record `llm_call` and `llm_fallback` now, with compact payloads that already match the later Phase 7 shape: member ID, nudge type, provider, success, latency, and fallback reason. Emit matching structured stdout logs so local review does not depend on querying SQLite manually. This step depends on step 6.
8. Phase H — Keep the API contract stable while making provenance visible. `GET /api/members/{member_id}/nudge` and `GET /api/coach/nudges` should continue returning the same structure, with `phrasing_source` simply reflecting the stored outcome. Add a small provenance label on the member card in `client/src/components/NudgeCard.tsx` and preserve the existing coach visibility in `client/src/components/CoachNudgesList.tsx`. Avoid broader UI changes, filters, or audit views in this phase. This step can run in parallel with step 7 once the backend field behavior is stable.
9. Phase I — Expand automated coverage around the new failure modes without introducing live-provider dependencies. Add unit tests for template selection, JSON parsing failures, blocked terms, length rejections, timeout handling, and provider errors. Add integration tests proving that the API returns `phrasing_source = "template"` with no key, preserves deterministic decisions on every fallback path, and can return `phrasing_source = "llm"` on a successful mocked provider response. This step depends on steps 4-8.
10. Phase J — Update reviewer-facing documentation in `README.md` and any relevant phase docs so local verification is obvious: how to run with and without `OPENAI_API_KEY`, what fallback behavior to expect, why support-risk stays deterministic, and how provenance appears in both member and coach views. This step depends on step 9.

**Verification**

1. Unit: verify template selection and fallback-reason mapping for `missing_key`, `timeout`, `provider_error`, `invalid_json`, and `validation_failure`.
2. Unit: verify blocked terms, missing fields, empty strings, and overlong `content` or `explanation` are all rejected before persistence.
3. Integration: confirm `GET /api/members/{member_id}/nudge` with no API key returns a successful active nudge with template copy and `phrasing_source = "template"`.
4. Integration: simulate timeout, malformed JSON, and validation failure and confirm the API still returns a valid deterministic nudge without changing `nudge_type`, `confidence`, or escalation behavior.
5. Integration: simulate a successful provider response and confirm the stored row upgrades to `phrasing_source = "llm"` exactly once for the newly created active nudge.
6. Integration: confirm repeated reads of an existing active nudge do not re-trigger the provider call and return the same nudge ID and stored phrasing.
7. Integration: confirm support-risk still returns `state = "escalated"` with no member-visible nudge and no regression in the coach escalation workflow.
8. UI/manual: verify the member card shows provenance for active nudges and the coach list still surfaces confidence, matched reason, and phrasing source clearly.

**Decisions**

- Included scope: optional OpenAI phrasing, deterministic fallback templates, strict output validation, minimal `llm_call` and `llm_fallback` audit hooks, and member-side provenance visibility.
- Excluded scope: LLM-driven decisioning, broader Phase 7 lifecycle instrumentation, new coach workflows, audit UI, multi-turn conversation, retrieval, or multi-provider abstraction.
- Recommended provider path: use the existing `httpx` dependency instead of adding a provider SDK so the prototype stays easy to inspect and control.
- Recommended orchestration point: keep phrasing inside the backend decision/persistence path so `GET /nudge` remains the single orchestration boundary.
- Recommended failure posture: do not retry provider calls within the same request; fail closed to deterministic templates immediately to protect latency and product reliability.
- Recommended support-risk posture: keep escalated outcomes deterministic and coach-facing only, even though the phase doc defines fallback copy for all nudge types.
