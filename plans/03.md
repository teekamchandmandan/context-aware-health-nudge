# Plan: Phase 03 — API Contracts

Implement the FastAPI API surface that exposes member decisions, records member actions, supports coach workflows, and accepts live member signals. Reuse the existing Phase 02 decision engine and SQLite schema where possible, and do not start frontend or phrasing work in this phase.

## Decisions

- **Thin route layer**: Keep route handlers small in `server/app/main.py` and move repeated SQL or transition logic into focused helper functions if the file starts to accumulate too much branching.
- **Reuse `evaluate_member()`**: `GET /api/members/{member_id}/nudge` should call the existing engine entrypoint so API idempotency follows the same active-nudge short-circuit already implemented in Phase 02.
- **Populate template `content` now**: Unlike the Phase 02 plan, Phase 03 should return member-safe deterministic template copy in `nudges.content` so Phase 04 can render nudges without waiting for Phase 06.
- **Expose only `explanation` publicly**: Keep `explanation_basis` as an internal engine concern unless a later phase explicitly needs it in the public contract.
- **`ask_for_help` escalation source**: Escalations created from a member action should use `source='member_action'` so they are distinguishable from low-confidence rule-engine escalations.
- **Signal intake stays append-only**: `POST /api/members/{member_id}/signals` persists the signal and returns it, but does not automatically trigger a fresh decision. The client should refetch `GET /nudge` explicitly.
- **No API expansion beyond spec**: Do not add auth, rich pagination, member discovery endpoints, or coach filtering in this phase.
- **Template content mapping**: Use a simple dict keyed by `nudge_type` to populate `content` at nudge creation time. Templates for the three current types:
  - `meal_guidance`: `"Consider a lighter, lower-carb option for your next meal."`
  - `weight_check_in`: `"It's been a few days since your last weigh-in — a quick check-in helps track your progress."`
  - `support_risk`: (not member-facing; nudge is escalated, so `content` stays `None`)
- **CORS**: Add permissive CORS middleware in Phase 03 so the Vite dev server can reach the API during Phase 04. Production tightening is out of scope.

## Phase A: API schema and validation

1. Add explicit request and response models for all planned endpoints:
   - Member nudge response with `state`, `member`, and nullable `nudge`
   - Action request (body with `action_type` enum) and action response (`nudge_id`, `action_type`, `nudge_status`, `recorded_at`)
   - Coach list responses with `items`, `limit`, and `count`
   - Signal intake request (`signal_type` + `payload` with conditional required fields) and stored signal response (`id`, `member_id`, `signal_type`, `payload`, `created_at`)

2. Use strict typed enums or equivalent validation for:
   - `action_type`: `act_now`, `dismiss`, `ask_for_help`
   - `signal_type`: `weight_logged`, `mood_logged`, `sleep_logged`

3. Enforce Phase 03 validation rules:
   - `weight_logged`: require `weight_lb`
   - `mood_logged`: require `mood`
   - `sleep_logged`: require `sleep_hours`
   - Unknown signal types or missing required payload fields return `422`

4. Standardize query validation for coach endpoints:
   - `limit` default `20`
   - Cap `limit` at `50`

## Phase B: Member-facing endpoints

5. Implement `GET /api/members/{member_id}/nudge`:
   - Check member existence first and return `404` before calling `evaluate_member()`, since the engine cannot distinguish a missing member from "no nudge"
   - Reuse `evaluate_member(conn, member_id)`
   - Translate engine return shapes into the API contract (the engine returns raw dicts with `nudge_id`/`escalation_id` for escalated state; the API must map these to `nudge: null`)
   - Attach member metadata (`id`, `name`) to every response variant
   - Keep the three explicit states: `active`, `no_nudge`, `escalated`
   - Repeated reads return the same active nudge until it becomes terminal

6. Update nudge persistence behavior to support the member contract:
   - Populate `content` with deterministic template copy when a nudge is created as `active`
   - Preserve `phrasing_source='template'`
   - Keep `explanation` factual and stable for coach visibility and later phrasing work

7. Implement `POST /api/members/{member_id}/signals`:
   - Return `404` if the member does not exist
   - Persist the normalized payload into `signals.payload_json`
   - Generate server-side IDs and UTC `created_at`
   - Return the stored signal record without triggering reevaluation

## Phase C: Action handling and transitions

8. Implement `POST /api/nudges/{nudge_id}/action`:
   - Return `404` if the nudge does not exist
   - Look up the nudge row to obtain `member_id` and current `status` (needed for escalation creation and terminal-state check)
   - Reject repeated actions on terminal nudges (`acted`, `dismissed`, `escalated`) with `409`
   - Persist one `nudge_actions` row per accepted action

9. Enforce action-to-status transitions:
   - `act_now` -> `acted`
   - `dismiss` -> `dismissed`
   - `ask_for_help` -> `escalated`

10. Add `ask_for_help` side effects:

- Create an escalation row automatically
- Set `source='member_action'`

11. Keep transition logic strict:

- Missing resources -> `404`
- Validation failures -> `422`
- Invalid state transitions -> `409`

## Phase D: Coach-facing endpoints

12. Implement `GET /api/coach/nudges`:

- Return recent nudges ordered by `created_at DESC`
- Include member name, content, explanation, matched reason, confidence, escalation recommendation, status, and latest user response
- Default to 20 items and cap at 50

13. Implement `GET /api/coach/escalations`:

- Return open escalations ordered by `created_at DESC`
- Include member name, reason, source, status, and timestamp
- Default to 20 items and cap at 50

14. Add query helpers if needed to keep handlers simple:

- Member lookup
- Nudge lookup
- Latest action lookup per nudge
- Coach nudge listing query
- Coach escalation listing query

## Phase E: Verification

15. Add API-level tests for:

- Repeated `GET /api/members/{member_id}/nudge` idempotency
- `404` for unknown member and unknown nudge IDs
- `409` for actions against terminal nudges
- `ask_for_help` escalation creation
- Valid and invalid payloads for all three supported signal types
- Coach endpoint ordering and limit behavior

16. Manual verification against seeded scenarios:

- `member_meal_01` returns an active meal guidance nudge
- `member_weight_01` returns an active weight check-in nudge
- `member_support_01` returns `state: "escalated"`
- Signal intake succeeds and a follow-up `GET /nudge` reflects the new context when appropriate
