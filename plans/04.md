# Plan: Phase 04 — Member Experience

Implement the member-facing vertical slice on top of the existing FastAPI contracts, with React Router for route structure and a modest UI/UX polish pass focused on trust, accessibility, and demo quality. Keep scope tight: one member page, one active nudge at a time, lightweight signal logging, explicit empty and error handling, and no backend expansion unless integration reveals a real contract gap.

## Decisions

- **Frontend-led phase**: Reuse the existing backend contract in `server/app/main.py` and `server/app/models.py`; do not add new Phase 04 endpoints unless a genuine mismatch appears during implementation.
- **Use React Router**: Add a real route for the member experience instead of keeping everything in a single ad hoc app component. This aligns better with later phases and keeps state URL-addressable.
- **Query-param member selection**: Use `/member?memberId=member_meal_01` and keep the selected member reflected in the URL so reviewers can deep-link or refresh safely.
- **Hardcoded seeded-member switcher**: Use the three known demo members instead of introducing a `GET /api/members` endpoint just for Phase 04.
- **Thin client data layer**: Add typed fetch helpers for nudge fetch, signal submission, and action submission. Do not introduce a global state library.
- **Server-confirmed refreshes**: After any signal submit or nudge action, refetch the current nudge state instead of relying on optimistic updates. Trust matters more than shaving off one round trip.
- **Mood input should match the backend contract**: Prefer a small three-state selector (`low`, `neutral`, `high`) plus optional note instead of a numeric slider, since the backend expects semantic mood values.
- **Meal logging stays structured first**: Prioritize `meal_type`, `carbs_g`, and optional `protein_g`. Photo capture is out of scope unless added only as a local preview enhancement that never replaces structured input.
- **Polish is included, but bounded**: Improve visual hierarchy, calm feedback, accessibility, and responsive behavior without adding extra product surface or dashboard complexity.
- **No scope bleed into later phases**: Coach workflows, LLM phrasing, observability expansion, auth, and broader dashboard behavior remain out of scope.
- **API base URL via env var**: Read the backend origin from `VITE_API_URL` (default `http://localhost:8000`) so the client never has a hardcoded host.
- **Member switcher as segmented buttons**: Three labeled buttons are cleaner than a select for exactly three options and keep the demo intent obvious.
- **`photo_attached` always `false`**: Always send `photo_attached: false` in meal signal payloads; do not implement file upload or local preview in Phase 04.
- **`meal_tag` as fallback for carbs_g**: If the reviewer does not enter a carbs value, fall back to a `meal_tag` string (the meal type name) so the backend validation rule `carbs_g OR meal_tag` is always satisfiable.
- **Post-action button state**: After any successful nudge action the action buttons should be hidden and replaced with a short inline confirmation. Re-fetching nudge state from the server drives the next render.
- **Mood input resolution**: Use a semantic three-state toggle (`low`, `neutral`, `high`) matching the backend contract. The numeric slider concept in the original spec doc is superseded by this decision. Update `docs/phase-04-member-experience.md` after implementation to reflect this.
- **Dev reset affordance**: Surface a small "Reset demo data" button in the member switcher area that calls `POST /debug/reset-seed` so reviewers can re-exercise scenarios without leaving the page or using curl.

   ```ts

   type ActionType = 'act_now' | 'dismiss' | 'ask_for_help';
   type SignalType = 'meal_logged' | 'weight_logged' | 'mood_logged';
   type MealType = 'breakfast' | 'lunch' | 'dinner' | 'snack';

     id: string;

   }
   interface NudgeDetail {
     id: string;

     content: string | null;

     confidence: number | null;
     escalation_recommended: boolean;
     status: string;
     phrasing_source: string;
     created_at: string;
   }
   interface MemberNudgeResponse {
     state: NudgeState;

     nudge_id: string;
     action_type: string;

     escalation_created: boolean;

   }
   interface SignalResponse {
     id: string;
     member_id: string;
     signal_type: string;
     payload: Record<string, unknown>;
     created_at: string;
   }
   ```

## Phase C: Build the member page interaction loop

8. Build `MemberSwitcher` with three segmented buttons (Alice / Bob / Carol), each updating the `memberId` query param. Include a small "Reset demo" button that calls `POST /debug/reset-seed` then refetches nudge state; keep it visually de-emphasised so it reads as a dev tool.

9. Build `QuickLog` with three signal paths matching the backend validation rules exactly:

   **Weight** — single number input; label "Weight (lb)"; requires `weight_lb > 0`.

   **Mood** — three-button toggle (`Low`, `Neutral`, `High`) mapping to `low`, `neutral`, `high`; optional note textarea; requires a selection before submit.

   **Meal** — `meal_type` select (`Breakfast`, `Lunch`, `Dinner`, `Snack`); `carbs_g` number input (optional); if `carbs_g` is empty, derive `meal_tag` from the selected `meal_type` to satisfy the backend requirement for `carbs_g OR meal_tag`; `protein_g` optional number input.

   Show frontend validation errors inline next to each field before sending; do not rely solely on the backend 422 response for user-facing validation feedback.

10. Build `NudgeCard` that shows:
    - recommendation content (`nudge.content`)
    - explanation under a "Why am I seeing this?" disclosure (separate visual treatment)
    - no confidence value in any member-visible element
    - three action buttons in separate visual weight: primary `Act now`, secondary `Dismiss`, tertiary `Ask for help`

11. Wire the three actions to `POST /api/nudges/{nudge_id}/action`. On success:
    - Hide action buttons immediately
    - Show a brief inline confirmation appropriate to each action:
      - `act_now` → "Got it. We will follow up."
      - `dismiss` → "Dismissed."
      - `ask_for_help` → "Flagged for your coach. We will be in touch."
    - Then refetch nudge state so the page reflects the new server truth

12. On API errors, distinguish by status code:
    - **Network / 5xx** — show the retryable error message from the spec; offer a "Try again" button
    - **409** — nudge already resolved; refetch state silently (do not show an error toast)
    - **422** — signal validation rejected; show inline field-level message adjacent to the offending input
    - **404** — should not occur with hardcoded member IDs; log to console, show generic retryable error

13. After any accepted signal or action, refetch the member nudge state so the interaction loop feels immediate and backend-driven.

## Risks to Watch

- **No reusable product UI exists yet**: Do not preserve Vite starter structure. Replace `App.css` entirely.
- **409 on repeated action**: If the page renders stale nudge state (e.g. after a fast double-click), the action endpoint returns 409. The fix is to refetch silently and re-render, not show an error toast.
- **`carbs_g OR meal_tag` validation**: The backend requires at least one of these. Always derive `meal_tag` from the selected `meal_type` so that a reviewer who skips the carbs field does not hit a 422.
- **Support scenario has no visible nudge card**: `member_support_01` returns `state: "escalated"` with `nudge: null`. The page must not crash or show a broken card; it should render only the escalated state message.
- **Local state complexity**: If `MemberPage` state grows beyond `{ nudge, loading, error }`, the refetch-driven loop has probably accumulated unnecessary complexity. Keep the state shape minimal.
- **`VITE_API_URL` in production builds**: The `.env.development` default is sufficient for Phase 04 local demo. Do not add build-time URL switching logic not needed for this phase.
- **Mood slider legacy**: The original spec doc still says "mood slider". Implementation should follow the semantic toggle decision in this plan; update the spec doc after implementation to avoid future confusion.
